# ADR-005：主核心关闭时使用隔离订阅探测运行时

- 状态：Proposed
- 日期：2026-07-23
- 关联：Issue #48；Issue #29；Issue #34；D-005、D-009、D-011

## 背景

节点目录、Provider 刷新和主动延迟测试原本都通过产品主核心的 loopback Controller 完成。主核心停止时，这些只读或诊断操作也一起不可用，用户必须先启动真实代理入口才能查看订阅节点或判断延迟。

直接解除前端禁用不能解决问题：节点目录和 provider member healthcheck 仍需要 Mihomo 解析订阅并提供受鉴权 Controller。同时，诊断需求不能成为修改系统代理、占用产品入口或放宽 Fail Closed 的理由。

## 决策

主核心 Controller 可用时继续使用主核心。主核心不可用时，节点目录、Provider 重试和主动延迟测试启动或复用应用自管的隔离订阅探测运行时。

隔离运行时必须满足以下约束：

- 入口和 Controller 只绑定随机 loopback 端口，并排除产品 entry、产品 Controller、`3666` 和 `6666`；
- 生成独立随机 Controller secret，调用方只持有内存中的鉴权客户端；
- 只加载已启用且受保护凭据可解析的订阅，selector 默认 `REJECT`，配置不存在 `DIRECT` 回退；
- 不修改 Windows 系统代理、TUN、DNS、路由或第三方客户端；
- 临时配置目录按私有路径加固，运行时按配置与订阅指纹复用，配置变化、退出或替换时只终止自身持有的 PID；
- Provider 加载和 healthcheck 继续使用已校验目标、受控超时、有界并发、取消和 generation 防护；
- UI 将“目录/测速 Controller 可用”和“主核心允许真实节点选择”建模为不同能力。隔离 selector 的临时 `now` 不作为主核心当前节点展示，真实选择保持禁用。

“主核心关闭时可测速”只表示不启动产品代理主路径，不表示系统中完全没有 Mihomo 子进程。UI 和文档必须明确隔离进程的存在与非接管边界。

## 替代方案

| 方案 | 结论 | 原因 |
|---|---|---|
| 保持现状，要求先启动主核心 | 拒绝 | 把诊断与启用真实代理路由耦合，核心关闭时节点页失去主要价值 |
| 前端直接下载并解析订阅 | 拒绝 | 重复协议解析与健康检查能力，扩大凭据和节点连接参数进入前端的范围 |
| 复用任意已监听的第三方 Controller | 拒绝 | 无法证明进程所有权、secret、配置或 selector 语义，违反第三方黑盒边界 |
| 隔离运行时允许 `DIRECT` 以便探测 | 拒绝 | 外部失败时可能静默直连，破坏 D-005 |
| 主核心关闭时仍允许持久化节点选择 | 拒绝 | 隔离 selector 不是产品路由权威状态，无法承诺启动后的真实选择 |

## 后果

正面结果是主核心关闭时仍可查看订阅节点并执行单节点或批量延迟测试，而产品入口、系统代理和真实节点选择保持不变。既有订阅探测运行时得到复用，不新增第二套订阅解析器，也不把敏感连接参数暴露给前端。

代价是一次诊断可能启动额外的受限 Mihomo 子进程并产生 Provider/healthcheck 外部请求。运行时必须承担临时端口、子进程、配置 generation、取消和清理生命周期；UI 也必须同时表达“可查看/测速”和“不可切换”的能力差异。

D-005、D-009 和 D-011 不变：无 `DIRECT` 回退，不持久化节点目录或测速结果，只管理能够证明所有权的应用子进程。

## 迁移与回滚

该决策不迁移配置 schema、Secret Store key 或 SQLite。新增的 `selection_ready` 只存在于 Tauri 到前端的运行时响应。

回滚时移除节点目录和测速命令的隔离回退，恢复主核心依赖即可；应用退出或新版本启动前必须终止自身持有的隔离子进程并清理临时目录。回滚不得停止端口相同但 PID/所有权不匹配的未知进程，也不得改变主核心和系统代理状态。
