# 本地客户端 A：16666 首轮兼容性验证

日期：2026-07-18

平台：Windows 10/11 x64

结论：部分通过，可以进入 Guardian 开发；稳定性和 UDP 仍需验证。

## 现场条件

| 项目 | 观测值 |
|---|---|
| 客户端 A 版本 | 2.2.4 |
| 客户端 A 核心 | `chaoshihuiCore.exe` |
| 配置 Mixed Port | `127.0.0.1:16666` |
| 系统代理开关 | 关闭 |
| TUN | 关闭 |
| Allow LAN | 关闭 |
| 自动启动代理引擎 | 关闭，需手动点击启动 |

## 验证结果

| 检查 | 结果 | 证据 |
|---|---|---|
| 端口可修改 | 通过 | 设置页与本地安全字段读取均显示 `16666` |
| 代理引擎未启动时 | 符合预期 | GUI/Core 进程存在，但 `16666` 无监听 |
| 手动启动后监听 | 通过 | `chaoshihuiCore.exe` 独占 `127.0.0.1:16666` |
| SOCKS5 TCP | 通过 | 经 `socks5h://127.0.0.1:16666` 请求返回 HTTP 204 |
| HTTP Mixed | 通过 | 经 `http://127.0.0.1:16666` 请求返回 HTTP 204 |
| 与现有 `6666` 并行 | 通过 | 两个核心分别监听 `16666` 与 `6666` |
| 不接管系统代理 | 通过 | Windows 系统代理仍指向 `127.0.0.1:6666` |
| 短时稳定性 | 未通过验收 | 5 个连续样本中 4 个成功、1 个 TLS 超时 |
| SOCKS5 UDP | 未验证 | 后续单独测试 |

## 重要解释

“客户端窗口已经打开”不代表 Mixed Port 已监听。该客户端保留了 FlClash 的运行状态模型：代理引擎启动后才创建监听。用于无人值守编排时，应启用客户端自身的自动运行能力，或由明确、稳定且经过验证的控制边界启动；第一版不采用 UI 自动化和未公开私有 IPC。
