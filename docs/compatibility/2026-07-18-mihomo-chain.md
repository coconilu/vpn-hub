# Mihomo 开发链路首轮验证

日期：2026-07-18
平台：Windows x64
结论：开发链路通过，未触碰受保护的 `6666`。第二轮验收已确认 Google、Gstatic、GitHub 和百度能够通过 `36666 → Mihomo → 16666` 完成 HTTPS 请求；特定目标仍可能受上游线路影响。

## 链路

```text
显式 curl → 127.0.0.1:36666 (Mihomo)
          → OUT-LOCAL-A
          → 127.0.0.1:16666 (Local client A core)
```

| 验证 | 结果 |
|---|---|
| Mihomo 监听 | `127.0.0.1:36666`，PID 与受控会话一致 |
| 国内 HTTPS 目标 | HTTP 200，约 81ms |
| 海外探测目标 | TLS 连接超时，与直接通过 `16666` 的结果一致 |
| Windows 系统代理 | 全程保持 `127.0.0.1:6666` |
| `6666` 所有者 | 测试前后均为原 Local client B core PID |
| 开发实例停止 | Ctrl+C 正常关闭，`36666` 随后释放 |
| 第三方客户端 | `16666` 与 `6666` 全程保持监听 |

## 第二轮外网端到端验收

验收时三个进程分别监听：

| 入口 | 进程 |
|---|---|
| `127.0.0.1:6666` | Local client B core |
| `127.0.0.1:16666` | Local client A core |
| `127.0.0.1:36666` | 本应用启动的官方 Mihomo v1.19.28 |

使用 `curl.exe --proxy socks5h://127.0.0.1:<port>` 对 `16666` 和 `36666` 进行同目标对照，结果如下：

| 目标 | `16666` | `36666` | 结论 |
|---|---:|---:|---|
| Google `generate_204` | HTTP 204，约 1.55s | HTTP 204，约 3.71s | 成功 |
| Gstatic `generate_204` | HTTP 204，约 1.70s | HTTP 204，约 1.88s | 成功 |
| GitHub | HTTP 200，约 3.42s | HTTP 200，约 2.29s | 成功 |
| 百度 | HTTP 200，约 69ms | HTTP 200，约 71ms | 成功 |
| Cloudflare `1.1.1.1` | TLS 超时 | TLS 超时 | 上游或目标兼容问题，不是中转独有故障 |

Google 另做三轮交替请求，`16666` 总耗时中位数约 1.75s，`36666` 约 1.71s；单次样本存在约 3.5s 抖动，当前证据不支持“本地中转固定增加约 1 秒延迟”的判断。

传输过程中，Windows `netstat` 捕获到 Mihomo 进程与 `127.0.0.1:16666` 的已建立连接；限速请求经 `36666` 实际接收了数据。开发配置的 `MASTER` 组仅包含 `OUT-LOCAL-A`，不存在 `DIRECT` 备用项，因此本轮流量没有绕过本地客户端 A。

测试完成后，`6666` 的监听进程和 Windows 系统代理 `127.0.0.1:6666` 均保持不变。

## 解释

首轮国内目标成功已经证明双层 SOCKS5/Mixed 代理链路成立；当时海外目标通过 `16666` 和 `36666` 同时失败，说明故障位于上游节点或线路，而不是开发编排层端口冲突。第二轮在上游状态改善后，两条路径均能访问 Google、Gstatic 和 GitHub，进一步确认 `36666` 可以稳定地把 TCP/HTTPS 流量交给 `16666`。Cloudflare 仍在两条路径上同时失败，因此健康判断后续需要使用多目标探测，不能把单个目标故障等同于整个出口断线。
